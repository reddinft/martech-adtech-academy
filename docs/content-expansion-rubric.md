# Content Expansion Rubric

Use this rubric to keep module depth consistent across the Academy.

## Tone
- Harvard cold-call, operator-practical, concise.
- Prefer decision prompts over generic discussion.
- Every prompt should force a tradeoff, owner, or metric.

## Density targets per module
- Outcomes: 4
- Theory: 6
- Practical: 5
- Discussion questions: 4
- Quiz: 4 to 5 questions, with at least one scenario or judgment item
- Real case exercise: 4 to 6 sequenced tasks

## Quality checks
- Keep architecture and types unchanged.
- Use concrete nouns, not vague learning language.
- Avoid subjective “best” claims without a metric.
- Write questions that can be answered from the module, but still require judgment.
- Keep one source-of-truth concept per module.
- Prefer operator actions, thresholds, and ownership over broad strategy talk.

## Real case sourcing
- Use 1 to 3 strong sources per module.
- Prefer first-party, standards-body, or reputable product documentation.
- Tighten source quality when an example depends on public facts.
- Do not pad with weak citations.

## BDD guidance
- Keep selectors stable.
- If copy changes, update tests to target role/structure, not brittle phrasing.
- Avoid assertions on subjective prose unless the text is intentionally fixed.
- Prefer checking key headings, labels, and system statuses.

## Review checklist
- [ ] Each module meets density targets.
- [ ] At least one quiz item per module is scenario-based.
- [ ] Each real case has sequenced tasks and source support.
- [ ] No type or architecture changes were introduced.
- [ ] Build and e2e tests pass.
